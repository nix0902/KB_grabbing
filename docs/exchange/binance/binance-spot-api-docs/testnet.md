<p><strong>Note:</strong> All features here will only apply to the <a href="https://testnet.binance.vision/" target="_blank" rel="noopener noreferrer">SPOT Testnet</a>.

</ul>

</ul>
<hr>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2026-02-09">2026-02-09<a class="hash-link" aria-label="Direct link to 2026-02-09" title="Direct link to 2026-02-09" href="/docs/binance-spot-api-docs/testnet#2026-02-09">​</a></h3>

</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2026-02-04">2026-02-04<a class="hash-link" aria-label="Direct link to 2026-02-04" title="Direct link to 2026-02-04" href="/docs/binance-spot-api-docs/testnet#2026-02-04">​</a></h3>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2025-12-15">2025-12-15<a class="hash-link" aria-label="Direct link to 2025-12-15" title="Direct link to 2025-12-15" href="/docs/binance-spot-api-docs/testnet#2025-12-15">​</a></h3>
<p><strong>Clarification Regarding UTF-8 Encoding:</strong></p>
<ul>
<li>In <a href="/docs/binance-spot-api-docs/testnet/fix-api">FIX</a>, <a href="https://developers.binance.com/docs/binance-spot-api-docs/testnet/rest-api/general-api-information" target="_blank" rel="noopener noreferrer">REST</a>, and <a href="https://developers.binance.com/docs/binance-spot-api-docs/testnet/websocket-api/general-api-information" target="_blank" rel="noopener noreferrer">WebSocket APIs</a>, if your request contains a symbol name containing non-ASCII characters, then the response may contain non-ASCII characters encoded in UTF-8.</li>
<li>In REST and WebSocket APIs, some endpoints/methods may return asset and/or symbol names containing non-ASCII characters encoded in UTF-8 even if the request did not contain non-ASCII characters.</li>
<li>In <a href="/docs/binance-spot-api-docs/testnet/web-socket-streams">WebSocket Streams</a>, if your request contains a symbol name containing non-ASCII characters, then the stream events may contain non-ASCII characters encoded in UTF-8.</li>
<li>In WebSocket Streams, <a href="/docs/binance-spot-api-docs/testnet/web-socket-streams#all-market-mini-tickers-stream">All Market Mini Tickers Stream</a> and <a href="/docs/binance-spot-api-docs/testnet/web-socket-streams#all-market-rolling-window-statistics-streams">All Market Rolling Window Statistics Streams</a> events may contain non-ASCII characters encoded in UTF-8.</li>
<li>In <a href="/docs/binance-spot-api-docs/testnet/sbe-market-data-streams">SBE Market Data Streams</a>, if your request contains a symbol name containing non-ASCII characters, then the stream events may contain non-ASCII characters encoded in UTF-8.</li>
<li><a href="/docs/binance-spot-api-docs/testnet/user-data-stream">UserDataStream events</a> may contain non-ASCII characters encoded in UTF-8 if you own or trade any assets or symbols whose names contain non-ASCII characters.</li>
<li>For full compatibility with Binance APIs, please ensure your code is designed to handle UTF-8-encoded strings.</li>
</ul>

<li>The previous behavior of <code>recvWindow</code> on FIX, REST, and WebSocket APIs will be augmented by an additional check.
<ul>
<li>To review, the existing behavior is:
<ul>
<li>If <code>timestamp</code> is greater than <code>serverTime</code> + 1 second at receipt of the request, the request is rejected. <li>Just before a request is forwarded to the Matching Engine, if the difference between <code>timestamp</code> and the current <code>serverTime</code> is greater than <code>recvWindow</code>, the request is rejected. <li><code>orderList.place.oto</code></li>
<li><code>orderList.place.otoco</code></li>
</ul>
</li>
</ul>
</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2024-04-04">2024-04-04<a class="hash-link" aria-label="Direct link to 2024-04-04" title="Direct link to 2024-04-04" href="/docs/binance-spot-api-docs/testnet#2024-04-04">​</a></h3>
<p>General changes:</p>
<ul>
<li>Symbol permission information in Exchange Information responses has moved from field <code>permissions</code> to field <code>permissionSets</code>.</li>
<li>Field <code>permissions</code> will be empty and will be removed in a future release.</li>
<li>Previously, <code>&quot;permissions&quot;:[&quot;SPOT&quot;,&quot;MARGIN&quot;]</code> meant that you could place an order on the symbol if your account had <code>SPOT</code> or <code>MARGIN</code> permissions. The equivalent is <code>&quot;permissionSets&quot;:[[&quot;SPOT&quot;,&quot;MARGIN&quot;]]</code>. (Note the extra set of square brackets.) Each array of permissions inside the <code>permissionSets</code> array is called a &quot;permission set&quot;.</li>
<li>Symbol permissions can now be more complex. <code>&quot;permissionSets&quot;:[[&quot;SPOT&quot;,&quot;MARGIN&quot;],[&quot;TRD_GRP_004&quot;,&quot;TRD_GRP_005&quot;]]</code> means that you may place an order on the symbol if your account has SPOT or MARGIN permissions <strong>and</strong> <code>TRD_GRP_004</code> or <code>TRD_GRP_005</code> permissions. There may be an arbitrary number of permission sets in a symbol&#x27;s <code>permissionSets</code>.</li>
<li><strong>The weight of the following requests has increased from 10 to 25</strong>:
<ul>
<li><code>GET /api/v3/trades</code></li>
<li><code>GET /api/v3/historicalTrades</code></li>
<li><code>trades.recent</code></li>
<li><code>trades.historical</code></li>
</ul>
</li>
</ul>
<p>REST API</p>
<ul>
<li>The <code>POST /api/v3/order/oco</code> endpoint is now deprecated on the REST API. You should use the new <code>POST /api/v3/orderList/oco</code> endpoint instead. Note that this new endpoint uses different parameters.</li>
<li><code>POST /api/v3/order/oco</code> has been removed from the Rest API documentation for SPOT Testnet.</li>
<li><code>otoAllowed</code> will now appear on <code>GET /api/v3/exchangeInfo</code>, that indicates if One-Triggers-the-Other (OTO) orders are supported on that symbol.</li>
</ul>
<p>WebSocket API</p>
<ul>
<li>The <code>orderList.place</code> request is now deprecated on the WebSocket API. You should now use the new <code>orderList.place.oco</code> request instead. Note that this new request uses different parameters.</li>
<li><code>orderList.place</code> has been removed from the WebSocket API documentation for SPOT Testnet.</li>
<li><code>otoAllowed</code> will now appear on <code>exchangeInfo</code>, that indicates if One-Triggers-the-Other (OTO) orders are supported on that symbol.</li>
</ul>
<p>SBE</p>
<ul>
<li>A new schema 2:0 <a href="https://github.com/binance/binance-spot-api-docs/blob/master/sbe/schemas/spot_2_0.xml" target="_blank" rel="noopener noreferrer">spot_2_0.xml</a> has been released for SPOT Testnet. The current schema, 1:0 <a href="https://github.com/binance/binance-spot-api-docs/blob/becd4d44a09d94821d2dc761ba9197aae8b495c3/sbe/schemas/spot_1_0.xml" target="_blank" rel="noopener noreferrer">spot_1_0.xml</a>, will thus be deprecated and retired from the Testnet APIs in 6 months as per our schema deprecation policy.</li>
<li>When using schema 1:0 on REST API or WebSocket API, group &quot;permissions&quot; in message &quot;ExchangeInfoResponse&quot; will always be empty. Upgrade to schema 2:0 to find permission information in group &quot;permissionSets&quot;. See General changes above for more details.</li>
<li>Responses for deprecated OCO requests are supported by both schema 1:0 and 2:0</li>
</ul>

<ul>
<li><code>GET /api/v3/account</code> has a new optional parameter <code>omitZeroBalances</code>, allowing to hide all zero balances</li>
<li><code>account.status</code> has a new optional parameter <code>omitZeroBalances</code> allowing to hide all zero balances.</li>
</ul>
<p>User Data Stream:</p>
<ul>
<li>New event <code>listenKeyExpired</code> is now emitted when a <code>listenKey</code> expires.</li>
