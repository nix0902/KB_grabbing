<h3 class="anchor anchorWithStickyNavbar_fMI7" id="connection-limits">Connection limits<a class="hash-link" aria-label="Direct link to Connection limits" title="Direct link to Connection limits" href="/docs/binance-spot-api-docs/websocket-api/rate-limits#connection-limits">​</a></h3>
<p>There is a limit of <strong>300 connections per attempt every 5 minutes</strong>.</p>
<p>The connection is per <strong>IP address</strong>.</p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="general-information-on-rate-limits">General information on rate limits<a class="hash-link" aria-label="Direct link to General information on rate limits" title="Direct link to General information on rate limits" href="/docs/binance-spot-api-docs/websocket-api/rate-limits#general-information-on-rate-limits">​</a></h3>
<ul>
<li>Current API rate limits can be queried using the <a href="/docs/binance-spot-api-docs/websocket-api/rate-limits#exchange-information"><code>exchangeInfo</code></a> request.</li>
<li>There are multiple rate limit types across multiple intervals.</li>
<li>Responses can indicate current rate limit status in the optional <code>rateLimits</code> field.</li>
<li>Requests fail with status <code>429</code> when unfilled order count or request rate limits are violated.</li>
</ul>

<p>Rate limits are accounted by intervals.</p>
<p>For example, a <code>1 MINUTE</code> interval starts every minute.
Request submitted at 00:01:23.456 counts towards the 00:01:00 minute&#x27;s limit.
Once the 00:02:00 minute starts, the count will reset to zero again.</p>
<p>Other intervals behave in a similar manner.
For example, <code>1 DAY</code> rate limit resets at 00:00 UTC every day,
and <code>10 SECOND</code> interval resets at 00, 10, 20... seconds of each minute.</p>
<p>APIs have multiple rate-limiting intervals.
If you exhaust a shorter interval but the longer interval still allows requests,
you will have to wait for the shorter interval to expire and reset.
If you exhaust a longer interval, you will have to wait for that interval to reset,
even if shorter rate limit count is zero.</p>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="ip-limits">IP limits<a class="hash-link" aria-label="Direct link to IP limits" title="Direct link to IP limits" href="/docs/binance-spot-api-docs/websocket-api/rate-limits#ip-limits">​</a></h3>
<ul>
<li>Every request has a certain <strong>weight</strong>, added to your limit as you perform requests.
<ul>
<li>The heavier the request (e.g. querying data from multiple symbols), the more weight the request will cost.</li>
<li>Connecting to WebSocket API costs 2 weight.</li>
</ul>
</li>
<li>Current weight usage is indicated by the <code>REQUEST_WEIGHT</code> rate limit type.</li>
<li>Use the <a href="/docs/binance-spot-api-docs/websocket-api/rate-limits#exchange-information"><code>exchangeInfo</code></a> request
to keep track of the current weight limits.</li>
<li>Weight is accumulated <strong>per IP address</strong> and is shared by all connections from that address.</li>
<li>If you go over the weight limit, requests fail with status <code>429</code>.
<ul>
<li>This status code indicates you should back off and stop spamming the API.</li>
<li>Rate-limited responses include a <code>retryAfter</code> field, indicating when you can retry the request.</li>
</ul>
</li>
<li><strong>Repeatedly violating rate limits and/or failing to back off after receiving 429s will result in an automated IP ban and you will be disconnected.</strong>
<ul>
<li>Requests from a banned IP address fail with status <code>418</code>.</li>
<li><code>retryAfter</code> field indicates the timestamp when the ban will be lifted.</li>
</ul>
</li>
<li>IP bans are tracked and <strong>scale in duration</strong> for repeat offenders, <strong>from 2 minutes to 3 days</strong>.</li>
</ul>
