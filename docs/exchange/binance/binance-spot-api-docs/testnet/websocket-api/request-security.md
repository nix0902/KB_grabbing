<li>Each method has a security type indicating required API key permissions, shown next to the method name (e.g., <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#place-new-order-trade">Place new order (TRADE)</a>).</li>
<li>If unspecified, the security type is <code>NONE</code>.</li>
<li>Except for <code>NONE</code>, all methods with a security type are considered <code>SIGNED</code> requests (i.e. including a <code>signature</code>), except for <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#user-data-stream-requests">listenKey management</a>.</li>
<li>Secure methods require a valid API key to be specified and authenticated.
<ul>
<li>API keys can be created on the <a href="https://testnet.binance.vision" target="_blank" rel="noopener noreferrer">SPOT Test Network</a> upon logging in with your Github account.</li>
<li><strong>Both API key and secret key are sensitive.</strong> Never share them with anyone.
If you notice unusual activity in your account, immediately revoke all the keys and contact Binance support.</li>
</ul>
</li>
<li>API keys can be configured to allow access only to certain types of secure methods.
<ul>
<li>For example, you can have an API key with <code>TRADE</code> permission for trading,
while using a separate API key with <code>USER_DATA</code> permission to monitor your order status.</li>
<li>By default, an API key cannot <code>TRADE</code>. You need to enable trading in API Management first.</li>
</ul>
</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="signed-request-security">SIGNED request security<a class="hash-link" aria-label="Direct link to SIGNED request security" title="Direct link to SIGNED request security" href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#signed-request-security">​</a></h3>
<ul>
<li><code>SIGNED</code> requests require an additional parameter: <code>signature</code>, authorizing the request.</li>
</ul>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="signature-case-sensitivity">Signature Case Sensitivity<a class="hash-link" aria-label="Direct link to Signature Case Sensitivity" title="Direct link to Signature Case Sensitivity" href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#signature-case-sensitivity">​</a></h4>
<ul>
<li><strong>HMAC:</strong> Signatures generated using HMAC are <strong>not case-sensitive</strong>. This means the signature string can be verified regardless of letter casing.</li>
<li><strong>RSA:</strong> Signatures generated using RSA are <strong>case-sensitive</strong>.</li>
<li><strong>Ed25519:</strong> Signatures generated using ED25519 are also <strong>case-sensitive</strong></li>
</ul>
<p>Please consult <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#signed-request-example-hmac">SIGNED request example (HMAC)</a>, <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#signed-request-example-rsa">SIGNED request example (RSA)</a>, and <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#signed-request-example-ed25519">SIGNED request example (Ed25519)</a> on how to compute signature, depending on which API key type you are using.</p>
<p><a id="timingsecurity"></a></p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="timing-security">Timing security<a class="hash-link" aria-label="Direct link to Timing security" title="Direct link to Timing security" href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#timing-security">​</a></h3>
<ul>
<li><code>SIGNED</code> requests also require a <code>timestamp</code> parameter which should be the current timestamp either in milliseconds or microseconds. (See <a href="/docs/binance-spot-api-docs/testnet/websocket-api/request-security#general-api-information">General API Information</a>)</li>
<li>An additional optional parameter, <code>recvWindow</code>, specifies for how long the request stays valid and may only be specified in milliseconds.
<ul>
<li><code>recvWindow</code> supports up to three decimal places of precision (e.g., 6000.346) so that microseconds may be specified.</li>
<li>If <code>recvWindow</code> is not sent, <strong>it defaults to 5000 milliseconds</strong>.</li>
<li>Maximum <code>recvWindow</code> is 60000 milliseconds.</li>
</ul>
</li>

<p><strong>Serious trading is about timing.</strong> Networks can be unstable and unreliable,
which can lead to requests taking varying amounts of time to reach the
servers. With <code>recvWindow</code>, you can specify that the request must be
processed within a certain number of milliseconds or be rejected by the
server.</p>
<p><strong>It is recommended to use a small <code>recvWindow</code> of 5000 or less!</strong></p>
