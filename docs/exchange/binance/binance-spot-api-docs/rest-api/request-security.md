content="binance-spot-api-docs">
  
<ul>
<li>Each endpoint has a security type indicating required API key permissions, shown next to the endpoint name (e.g., <a href="/docs/binance-spot-api-docs/rest-api/request-security#place-new-order-trade">New order (TRADE)</a>).</li>
<li>If unspecified, the security type is <code>NONE</code>.</li>
<li>Except for <code>NONE</code>, all endpoints with a security type are considered <code>SIGNED</code> requests (i.e. including a <code>signature</code>), except for <a href="/docs/binance-spot-api-docs/rest-api/request-security#user-data-stream-requests">listenKey management</a>.</li>
<li>Secure endpoints require a valid API key to be specified and authenticated.
<ul>
<li>API keys can be created on the <a href="https://www.binance.com/en/support/faq/360002502072" target="_blank" rel="noopener noreferrer">API Management</a> page of your Binance account.</li>
<li><strong>Both API key and secret key are sensitive.</strong> Never share them with anyone.
If you notice unusual activity in your account, immediately revoke all the keys and contact Binance support.</li>
</ul>
</li>
<li>API keys can be configured to allow access only to certain types of secure endpoints.
<ul>
<li>For example, you can have an API key with <code>TRADE</code> permission for trading,
while using a separate API key with <code>USER_DATA</code> permission to monitor your order status.</li>
<li>By default, an API key cannot <code>TRADE</code>. You need to enable trading in API Management first.</li>
</ul>
</li>
</ul>

























<table><thead><tr><th>Security type</th><th>Description</th></tr></thead><tbody><tr><td><code>NONE</code></td><td>Public market data</td></tr><tr><td><code>TRADE</code></td><td>Trading on the exchange, placing and canceling orders</td></tr><tr><td><code>USER_DATA</code></td><td>Private account information, such as order status and your trading history</td></tr><tr><td><code>USER_STREAM</code></td><td>Managing User Data Stream subscriptions</td></tr></tbody></table>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="signed-endpoint-security">SIGNED Endpoint security<a class="hash-link" aria-label="Direct link to SIGNED Endpoint security" title="Direct link to SIGNED Endpoint security" href="/docs/binance-spot-api-docs/rest-api/request-security#signed-endpoint-security">​</a></h3>
<ul>
<li><code>SIGNED</code> endpoints require an additional parameter, <code>signature</code>, to be sent in the <code>query string</code> or <code>request body</code>.</li>
</ul>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="signature-case-sensitivity">Signature Case Sensitivity<a class="hash-link" aria-label="Direct link to Signature Case Sensitivity" title="Direct link to Signature Case Sensitivity" href="/docs/binance-spot-api-docs/rest-api/request-security#signature-case-sensitivity">​</a></h4>
<ul>
<li><strong>HMAC:</strong> Signatures generated using HMAC are <strong>not case-sensitive</strong>. This means the signature string can be verified regardless of letter casing.</li>
<li><strong>RSA:</strong> Signatures generated using RSA are <strong>case-sensitive</strong>.</li>
<li><strong>Ed25519:</strong> Signatures generated using Ed25519 are also <strong>case-sensitive</strong></li>
</ul>
<p>Please consult <a href="/docs/binance-spot-api-docs/rest-api/request-security#hmac-keys">SIGNED request example (HMAC)</a>, <a href="/docs/binance-spot-api-docs/rest-api/request-security#rsa-keys">SIGNED request example (RSA)</a>, and <a href="/docs/binance-spot-api-docs/rest-api/request-security#ed25519-keys">SIGNED request example (Ed25519)</a> on how to compute signature, depending on which API key type you are using.</p>
<p><a id="timingsecurity"></a></p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="timing-security">Timing security<a class="hash-link" aria-label="Direct link to Timing security" title="Direct link to Timing security" href="/docs/binance-spot-api-docs/rest-api/request-security#timing-security">​</a></h3>
<ul>
<li><code>SIGNED</code> requests also require a <code>timestamp</code> parameter which should be the current timestamp either in milliseconds or microseconds. (See <a href="/docs/binance-spot-api-docs/rest-api/request-security#general-api-information">General API Information</a>)</li>
<li>An additional optional parameter, <code>recvWindow</code>, specifies for how long the request stays valid and may only be specified in milliseconds.
<ul>
<li><code>recvWindow</code> supports up to three decimal places of precision (e.g., 6000.346) so that microseconds may be specified.</li>

<h4 class="anchor anchorWithStickyNavbar_fMI7" id="rsa-keys">RSA Keys<a class="hash-link" aria-label="Direct link to RSA Keys" title="Direct link to RSA Keys" href="/docs/binance-spot-api-docs/rest-api/request-security#rsa-keys">​</a></h4>
<p>The signature payload of your request is the query string concatenated without separator to the HTTP body. Any non-ASCII character must be percent-encoded before signing.</p>
<p>To get your API key, you need to upload your RSA Public Key to your account and a corresponding API key will be provided for you.</p>
<p>Only <code>PKCS#8</code> keys are supported.</p>
<p>There is one example with a symbol name comprised entirely of ASCII characters and one example with a symbol name containing non-ASCII characters.</p>
