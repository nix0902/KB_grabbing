<h1>WebSocket Streams for Binance</h1>
<h2 class="anchor anchorWithStickyNavbar_fMI7" id="general-wss-information">General WSS information<a class="hash-link" aria-label="Direct link to General WSS information" title="Direct link to General WSS information" href="/docs/binance-spot-api-docs/web-socket-streams#general-wss-information">​</a></h2>
<ul>
<li>The base endpoint is: <strong>wss://stream.binance.com:9443</strong> or <strong>wss://stream.binance.com:443</strong>.</li>
<li>Streams can be accessed either in a single raw stream or in a combined stream.</li>
<li>Raw streams are accessed at <strong>/ws/&lt;streamName&gt;</strong></li>
<li>Combined streams are accessed at <strong>/stream?streams=&lt;streamName1&gt;/&lt;streamName2&gt;/&lt;streamName3&gt;</strong></li>
<li>Combined stream events are wrapped as follows: <strong>{&quot;stream&quot;:&quot;&lt;streamName&gt;&quot;,&quot;data&quot;:&lt;rawPayload&gt;}</strong></li>
<li>All symbols for streams are <strong>lowercase</strong></li>
<li>A single connection to <strong>stream.binance.com</strong> is only valid for 24 hours; expect to be disconnected at the 24 hour mark</li>
<li>The WebSocket server will send a <code>ping frame</code> every 20 seconds.
<ul>
<li>If the WebSocket server does not receive a <code>pong frame</code> back from the connection within a minute the connection will be disconnected.</li>
<li>When you receive a ping, you must send a pong with a copy of ping&#x27;s payload as soon as possible.</li>
<li>Unsolicited <code>pong frames</code> are allowed, but will not prevent disconnection. <strong>It is recommended that the payload for these pong frames are empty.</strong></li>
</ul>
</li>
<li>The base endpoint <strong>wss://data-stream.binance.vision</strong> can be subscribed to receive <strong>only</strong> market data messages. <br> User data stream is <strong>NOT</strong> available from this URL.</li>
<li>All time and timestamp related fields are <strong>milliseconds by default</strong>. To receive the information in microseconds, please add the parameter <code>timeUnit=MICROSECOND or timeUnit=microsecond</code> in the URL.
<ul>
<li>For example: <code>/stream?streams=btcusdt@trade&amp;timeUnit=MICROSECOND</code></li>
</ul>
</li>
<li>If your request contains a symbol name containing non-ASCII characters, then the stream events may contain non-ASCII characters encoded in UTF-8.</li>
<li>[All Market Mini Tickers Stream](#all-market-mini-tickers-stream and <a href="/docs/binance-spot-api-docs/web-socket-streams#all-market-rolling-window-statistics-streams">All Market Rolling Window Statistics Streams</a> events may contain non-ASCII characters encoded in UTF-8.</li>
</ul>
<h2 class="anchor anchorWithStickyNavbar_fMI7" id="websocket-limits">WebSocket Limits<a class="hash-link" aria-label="Direct link to WebSocket Limits" title="Direct link to WebSocket Limits" href="/docs/binance-spot-api-docs/web-socket-streams#websocket-limits">​</a></h2>
<ul>
<li>WebSocket connections have a limit of 5 incoming messages per second. A message is considered:
<ul>
<li>A PING frame</li>
<li>A PONG frame</li>
<li>A JSON controlled message (e.g. subscribe, unsubscribe)</li>
</ul>
</li>
<li>A connection that goes beyond the limit will be disconnected; IPs that are repeatedly disconnected may be banned.</li>
<li>A single connection can listen to a maximum of 1024 streams.</li>
<li>There is a limit of <strong>300 connections per attempt every 5 minutes per IP</strong>.</li>
</ul>
<h2 class="anchor anchorWithStickyNavbar_fMI7" id="live-subscribingunsubscribing-to-streams">Live Subscribing/Unsubscribing to streams<a class="hash-link" aria-label="Direct link to Live Subscribing/Unsubscribing to streams" title="Direct link to Live Subscribing/Unsubscribing to streams" href="/docs/binance-spot-api-docs/web-socket-streams#live-subscribingunsubscribing-to-streams">​</a></h2>
<ul>
<li>The following data can be sent through the WebSocket instance in order to subscribe/unsubscribe from streams. Examples can be seen below.</li>
<li>The <code>id</code> is used as an identifier to uniquely identify the messages going back and forth. The following formats are accepted:
<ul>
<li>64-bit signed integer</li>
<li>alphanumeric strings; max length 36</li>
<li><code>null</code></li>
</ul>
</li>
<li>In the response, if the <code>result</code> received is <code>null</code> this means the request sent was a success for non-query requests (e.g. Subscribing/Unsubscribing).</li>
</ul>

<h2 class="anchor anchorWithStickyNavbar_fMI7" id="klinecandlestick-streams-with-timezone-offset">Kline/Candlestick Streams with timezone offset<a class="hash-link" aria-label="Direct link to Kline/Candlestick Streams with timezone offset" title="Direct link to Kline/Candlestick Streams with timezone offset" href="/docs/binance-spot-api-docs/web-socket-streams#klinecandlestick-streams-with-timezone-offset">​</a></h2>

<ul>
<li>Kline intervals open and close in the <code>UTC+8</code> timezone. For example the <code>1d</code> klines will open at the beginning of the <code>UTC+8</code> day, and close at the end of the <code>UTC+8</code> day.</li>

<h2 class="anchor anchorWithStickyNavbar_fMI7" id="how-to-manage-a-local-order-book-correctly">How to manage a local order book correctly<a class="hash-link" aria-label="Direct link to How to manage a local order book correctly" title="Direct link to How to manage a local order book correctly" href="/docs/binance-spot-api-docs/web-socket-streams#how-to-manage-a-local-order-book-correctly">​</a></h2>
<ol>
<li>Open a WebSocket connection to <code>wss://stream.binance.com:9443/ws/bnbbtc@depth</code>.</li>
<li>Buffer the events received from the stream. Note the <code>U</code> of the first event you received.</li>
<li>Get a depth snapshot from <code>https://api.binance.com/api/v3/depth?symbol=BNBBTC&amp;limit=5000</code>.</li>
<li>If the <code>lastUpdateId</code> from the snapshot is strictly less than the <code>U</code> from step 2, go back to step 3.</li>
<li>In the buffered events, discard any event where <code>u</code> is &lt;= <code>lastUpdateId</code> of the snapshot. The first buffered event should now have <code>lastUpdateId</code> within its <code>[U;u]</code> range.</li>
<li>Set your local order book to the snapshot. Its update ID is <code>lastUpdateId</code>.</li>
<li>Apply the update procedure below to all buffered events, and then to all subsequent events received.</li>
</ol>
<p>To apply an event to your local order book, follow this update procedure:</p>
<ol>
<li>Decide whether the update event can be applied:
<ul>
<li>If the event last update ID (<code>u</code>) is less than the update ID of your local order book,
ignore the event.</li>
<li>If the event first update ID (<code>U</code>) is greater than the update ID of your local order book + 1,
you have missed some events. <br> Discard your local order book and restart the process from the beginning.</li>
<li>Normally, <code>U</code> of the next event is equal to <code>u + 1</code> of the previous event.</li>
</ul>
</li>
<li>For each price level in bids (<code>b</code>) and asks (<code>a</code>), set the new quantity in the order book:
<ul>
<li>If the price level does not exist in the order book, insert it with new quantity.</li>
<li>If the quantity is zero, remove the price level from the order book.</li>
</ul>
</li>
<li>Set the order book update ID to the last update ID (<code>u</code>) in the processed event.</li>
</ol>
<blockquote>
<p>[!NOTE]
Since depth snapshots retrieved from the API have a limit on the number of price levels (5000 on each side maximum), you won&#x27;t learn the quantities for the levels outside of the initial snapshot unless they change. <br>
So be careful when using the information for those levels, since they might not reflect the full view of the order book. <br>
However, for most use cases, seeing 5000 levels on each side is enough to understand the market and trade effectively.</p>
