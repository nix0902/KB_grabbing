<h1>Simple Binary Encoding (SBE) FAQ</h1>
<p>The goal of this document is to explain:</p>
<ul>
<li>How to receive SBE responses in the SPOT API.</li>
<li>How to decode SBE responses.</li>
</ul>
<p>SBE is a serialization format used for low-latency.</p>
<p>This implementation is based on the FIX SBE specification.</p>
<ul>
<li><a href="https://github.com/FIXTradingCommunity/fix-simple-binary-encoding" target="_blank" rel="noopener noreferrer">GitHub repository</a></li>
<li><a href="https://www.fixtrading.org/standards/sbe-online" target="_blank" rel="noopener noreferrer">HTML document</a></li>
</ul>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="how-to-get-an-sbe-response">How to get an SBE response<a class="hash-link" aria-label="Direct link to How to get an SBE response" title="Direct link to How to get an SBE response" href="/docs/binance-spot-api-docs/faqs/sbe_faq#how-to-get-an-sbe-response">​</a></h3>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="rest-api">REST API<a class="hash-link" aria-label="Direct link to REST API" title="Direct link to REST API" href="/docs/binance-spot-api-docs/faqs/sbe_faq#rest-api">​</a></h4>
<ul>
<li>The <code>Accept</code> header must include <code>application/sbe</code>.</li>
<li>Provide the schema ID and version in the <code>X-MBX-SBE</code> header as <code>&lt;ID&gt;:&lt;VERSION&gt;</code>.</li>
</ul>
<p>Sample request (REST):</p>
<div class="codeBlockContainer_zbXh theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_RFcP"><pre tabindex="0" class="prism-code language-text codeBlock_hldk thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_aHhF"><span class="token-line" style="color:#393A34"><span class="token plain">curl -sX GET -H &quot;Accept: application/sbe&quot; -H &quot;X-MBX-SBE: 1:0&quot; &#x27;https://api.binance.com/api/v3/exchangeInfo?symbol=BTCUSDT&#x27;</span><br></span></code></pre><div class="buttonGroup_oDVH"><button type="button" aria-label="Copy code to clipboard" title="Copy" class="clean-btn"><span class="copyButtonIcons_hTZm" aria-hidden="true"><svg viewBox="0 0 24 24" class="copyButtonIcon_xA0c"><path fill="currentColor" d="M19,21H8V7H19M19,5H8A2,2 0 0,0 6,7V21A2,2 0 0,0 8,23H19A2,2 0 0,0 21,21V7A2,2 0 0,0 19,5M16,1H4A2,2 0 0,0 2,3V17H4V3H16V1Z"></path></svg><svg viewBox="0 0 24 24" class="copyButtonSuccessIcon_ZIiU"><path fill="currentColor" d="M21,7L9,19L3.5,13.5L4.91,12.09L9,16.17L19.59,5.59L21,7Z"></path></svg></span></button></div></div></div>
<p><strong>Notes:</strong></p>
<ul>
<li>If you provide only <code>application/sbe</code> in the Accept header:
<ul>
<li>If SBE is not enabled in the exchange, you will receive an HTTP <strong>406 Not Acceptable</strong>.</li>
<li>If the <code>&lt;ID&gt;:&lt;VERSION&gt;</code> provided in the <code>X-MBX-SBE</code> header is malformed or invalid, the response will be an SBE-encoded error.</li>
<li>If the <code>X-MBX-SBE</code> header is missing, the response will be an SBE-encoded error.</li>
</ul>
</li>
<li>If you provide both <code>application/sbe</code> and <code>application/json</code> in the Accept header:
<ul>
<li>If SBE is not enabled in the exchange, the response will fall back to JSON.</li>
<li>If the <code>&lt;ID&gt;:&lt;VERSION&gt;</code> provided in the <code>X-MBX-SBE</code> header is malformed or invalid, the response will fall back to JSON.</li>
<li>If the <code>X-MBX-SBE</code> header is missing, the response will fall back to JSON.</li>
</ul>
</li>
</ul>

<ul>
<li>In the connection URL, add <code>responseFormat=sbe</code>.</li>
<li>Provide the schema ID and version in the parameters <code>sbeSchemaId=&lt;SCHEMA_ID&gt;</code> and <code>sbeSchemaVersion=&lt;SCHEMA_VERSION&gt;</code> respectively.</li>
</ul>

<p><strong>Notes:</strong></p>
<ul>
<li>If you provide only <code>responseFormat=sbe</code> in the connection URL:
<ul>
<li>If SBE is not enabled in the exchange, the response will be HTTP 400.</li>
<li>If the <code>sbeSchemaId=&lt;SCHEMA_ID&gt;</code> or <code>sbeSchemaVersion=&lt;SCHEMA_VERSION&gt;</code> are malformed or invalid, the response will be HTTP 400.</li>
</ul>
</li>
<li>If you provide both <code>responseFormat=sbe</code> and <code>responseFormat=json</code>, the response will be HTTP 400.</li>
<li>All error responses during the HTTP handshake are encoded as JSON with the <code>Content-Type</code> header set to <code>application/json;charset=UTF-8</code>.</li>
<li>Once a WebSocket session has been successfully established with SBE enabled, all method responses within that session are encoded in SBE, even in the event SBE becomes disabled.
<ul>
<li>This means that if SBE is disabled while your WebSocket connection is active, you will receive an SBE-encoded &quot;SBE is not enabled&quot; error in response to any subsequent request.</li>
</ul>
</li>
<li>As of writing, we do not recommend using <code>websocat</code> to send any request as we have observed issues in how it decodes binary frames. The sample above is only used for reference to show the URL to get an SBE response.</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="regarding-legacy-support">Regarding Legacy support<a class="hash-link" aria-label="Direct link to Regarding Legacy support" title="Direct link to Regarding Legacy support" href="/docs/binance-spot-api-docs/faqs/sbe_faq#regarding-legacy-support">​</a></h3>
<ul>
<li>SBE schemas are versioned via two XML attributes, <code>id</code> and <code>version</code>.
<ul>
<li><code>id</code> is incremented when a breaking change is introduced. When this occurs, <code>version</code> is reset to 0.</li>
<li><code>version</code> is incremented when a non-breaking change is introduced. When this occurs, <code>id</code> is not modified.</li>
</ul>
</li>
<li>When a new schema is live the old schema becomes deprecated. <strong>Deprecation occurs even when the new schema only introduces non-breaking changes.</strong></li>
<li>A deprecated schema will be supported <strong>for at least 6 months after deprecation</strong>. <br>For example given this hypothetical timeline:
<ul>
<li>January 3024: Schema id 1 version 0 is released. This is the first version so this is usable once SBE is enabled in the exchange.</li>
<li>March 3024: Schema id 1 version 1 is released. This schema introduces a non-breaking change.
<ul>
<li>Schema id 1 version 0 is deprecated, but can still be used for at least another 6 months.</li>
</ul>
</li>
<li>August 3024: Schema id 2 version 0 is released. This schema introduces a breaking change.
<ul>
<li>Schema id 1 version 0 is deprecated, but can still be used for at least another 1 month.</li>
<li>Schema id 1 version 1 is deprecated, but can still be used for at least another 6 months.</li>
</ul>
</li>
<li>September 3024: 6 months have passed since the release of Schema id 1 version 1.
<ul>
<li>Schema id 1 version 0 is retired.</li>
</ul>
</li>
<li>February 3025: Schema id 2 version 1 is released. This schema introduces a non-breaking change.
<ul>
<li>Schema id 1 version 1 is retired.</li>
<li>Schema id 2 version 0 is deprecated, but can still be used for at least another 6 months.</li>
</ul>
</li>
</ul>
</li>
<li>For REST API requests specifying a deprecated <code>&lt;ID&gt;:&lt;VERSION&gt;</code> in their <code>X-MBX-SBE</code> header:
<ul>
<li>the HTTP responses will contain a <code>X-MBX-SBE-DEPRECATED</code> header</li>
<li>the SBE responses will be encoded in the highest compatible schema
<ul>
<li>For example, as of 2025-08-27, requests for <code>X-MBX-SBE: 3:0</code> will receive responses encoded in schema <code>3:1</code>. An SBE decoder for schema <code>3:0</code> is expected to decode schema <code>3:1</code> gracefully as detailed in the <a href="https://www.fixtrading.org/standards/sbe-online/#schema-extension-mechanism" target="_blank" rel="noopener noreferrer">FIX SBE Specification</a>.</li>
</ul>
</li>
</ul>
</li>
<li>For WebSocket API connections specifying a deprecated <code>sbeSchemaId</code> and <code>sbeSchemaVersion</code> in their connection URL:
<ul>
<li>the field <code>sbeSchemaIdVersionDeprecated</code> will be set to <code>true</code> in all <code>WebSocketResponse</code> SBE messages</li>
<li>all SBE responses will be encoded in the highest compatible schema
<ul>
<li>For example, as of 2025-08-27, requests for <code>sbeSchemaId=3&amp;sbeSchemaVersion=0</code> will receive responses encoded in schema <code>3:1</code>. An SBE decoder for schema <code>3:0</code> is expected to decode schema <code>3:1</code> gracefully as detailed in the <a href="https://www.fixtrading.org/standards/sbe-online/#schema-extension-mechanism" target="_blank" rel="noopener noreferrer">FIX SBE Specification</a>.</li>
</ul>
</li>
</ul>
</li>
<li>For FIX API, when an SBE request message header specifies a deprecated <code>schemaId</code> and <code>version</code>:
<ul>
<li>the field <code>sbeSchemaIdVersionDeprecated</code> will be set to <code>true</code> in the <code>LogonAck</code> message</li>
<li>all SBE response messages will be encoded using the highest schema version for the provided <code>schemaId</code></li>
</ul>
</li>
<li>Requests specifying a retired schemaId/version will fail with HTTP 400 (REST &amp; WebSocket) or reject message (FIX API) .</li>
<li>In SBE Schema <a href="https://github.com/binance/binance-spot-api-docs/blob/master/sbe/schemas/spot_3_0.xml" target="_blank" rel="noopener noreferrer">3:0</a>, a <code>validValue</code> named <code>NonRepresentable</code> was added to each <code>enum</code>. Receipt of this value indicates that additional data is available when using the latest schema.</li>
<li>In SBE Schema <a href="https://github.com/binance/binance-spot-api-docs/blob/master/sbe/schemas/spot_3_1.xml" target="_blank" rel="noopener noreferrer">3:1</a>, a message named <code>NonRepresentableMessage</code> was added. Receipt of this message indicates that additional data is available when using the latest schema. This message may be received as a top-level message or embedded in a <code>data</code> field when the <code>data</code> field&#x27;s <code>type</code> is <code>messageData</code>, <code>messageData8</code>, <code>messageData16</code>, <code>optionalMessageData</code>, or <code>optionalMessageData16</code>.</li>
<li>JSON file regarding the schema life-cycle with the dates of the latest, deprecated, and retired schemas for both the live exchange and SPOT Testnet will be saved in this repository <a href="https://github.com/binance/binance-spot-api-docs/tree/master/sbe/schemas" target="_blank" rel="noopener noreferrer">here</a>. <br> Below is an example JSON based on the hypothetical timeline above:</li>

<h4 class="anchor anchorWithStickyNavbar_fMI7" id="decimal-field-encoding">Decimal field encoding<a class="hash-link" aria-label="Direct link to Decimal field encoding" title="Direct link to Decimal field encoding" href="/docs/binance-spot-api-docs/faqs/sbe_faq#decimal-field-encoding">​</a></h4>
<p>Unlike the FIX SBE specification, decimal fields have their mantissa and exponent fields encoded separately as primitive fields in order to minimize payload size and the number of encoded fields within messages.</p>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="timestamp-field-encoding">Timestamp field encoding<a class="hash-link" aria-label="Direct link to Timestamp field encoding" title="Direct link to Timestamp field encoding" href="/docs/binance-spot-api-docs/faqs/sbe_faq#timestamp-field-encoding">​</a></h4>
<p>Timestamps in SBE responses are in microseconds. This differs from JSON responses, which contain millisecond timestamps by default.</p>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="custom-field-attributes-in-the-schema-file">Custom field attributes in the schema file<a class="hash-link" aria-label="Direct link to Custom field attributes in the schema file" title="Direct link to Custom field attributes in the schema file" href="/docs/binance-spot-api-docs/faqs/sbe_faq#custom-field-attributes-in-the-schema-file">​</a></h4>
<p>A few field attributes prefixed with <code>mbx:</code> were added to the schema file for documentation purposes:</p>
<ul>
<li><code>mbx:exponent</code>: Points to the exponent field corresponding to the mantissa field</li>
<li><code>mbx:jsonPath</code>: Contains the name of the equivalent field in the JSON response</li>
<li><code>mbx:jsonValue</code>: Contains the name of the equivalent ENUM value in the JSON response</li>
