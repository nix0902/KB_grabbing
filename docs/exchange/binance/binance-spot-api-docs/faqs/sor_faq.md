<h1>Smart Order Routing (SOR)</h1>
<p><strong>Disclaimer:</strong></p>
<ul>
<li>The symbols and values used here are fictional, and do not imply anything about the actual setup on the live exchange.</li>
<li>For simplicity, the examples in this document do not include commission.</li>
</ul>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="what-is-smart-order-routing-sor">What is Smart Order Routing (SOR)?<a class="hash-link" aria-label="Direct link to What is Smart Order Routing (SOR)?" title="Direct link to What is Smart Order Routing (SOR)?" href="/docs/binance-spot-api-docs/faqs/sor_faq#what-is-smart-order-routing-sor">​</a></h3>
<p><strong>Smart Order Routing (SOR)</strong> allows you to potentially get better liquidity by filling an order with liquidity from other order books with the same base asset and interchangeable quote assets. <strong>Interchangeable quote assets</strong> are quote assets with fixed 1 to 1 exchange rate, such as stablecoins pegged to the same fiat currency.</p>
<p>Note that even though the quote assets are interchangeable, when selling the base asset you will always receive the quote asset of the symbol in your order.</p>
<p>When you place an order using SOR, it goes through the eligible order books, looks for best price levels for each order book in that SOR configuration, and takes from those books if possible.</p>
<p><strong>Note:</strong> If the order using SOR cannot fully fill based on the eligible order books&#x27; liquidity, <code>LIMIT IOC</code> or <code>MARKET</code> orders will immediately expire, while <code>LIMIT GTC</code> orders will place the remaining quantity on the order book you originally submitted the order to.</p>
<p><strong>Example 1</strong></p>
<p>Let&#x27;s consider a SOR configuration containing the symbols <code>BTCUSDT</code>, <code>BTCUSDC</code> and <code>BTCUSDP</code>, and the following <code>ASK</code> (<code>SELL</code> side) order books for those symbols:</p>

<p>On the WebSocket API, the request is <code>sor.order.place</code>.</p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="in-the-api-response-theres-a-field-called-workingfloor-what-does-that-field-mean">In the API response, there&#x27;s a field called <code>workingFloor</code>. What does that field mean?<a class="hash-link" aria-label="Direct link to in-the-api-response-theres-a-field-called-workingfloor-what-does-that-field-mean" title="Direct link to in-the-api-response-theres-a-field-called-workingfloor-what-does-that-field-mean" href="/docs/binance-spot-api-docs/faqs/sor_faq#in-the-api-response-theres-a-field-called-workingfloor-what-does-that-field-mean">​</a></h3>
<p>This is a term used to determine where the order&#x27;s last activity occurred (filling, expiring, or being placed as new, etc.).</p>
<p>If the <code>workingFloor</code> is <code>SOR</code>, this means your order interacted with other eligible order books in the SOR configuration.</p>
<p>If the <code>workingFloor</code> is <code>EXCHANGE</code>, this means your order interacted on the order book that you sent that order to.</p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="in-the-api-response-fills-contain-fields-matchtype-and-allocid-what-do-they-mean">In the API response, <code>fills</code> contain fields <code>matchType</code> and <code>allocId</code>. What do they mean?<a class="hash-link" aria-label="Direct link to in-the-api-response-fills-contain-fields-matchtype-and-allocid-what-do-they-mean" title="Direct link to in-the-api-response-fills-contain-fields-matchtype-and-allocid-what-do-they-mean" href="/docs/binance-spot-api-docs/faqs/sor_faq#in-the-api-response-fills-contain-fields-matchtype-and-allocid-what-do-they-mean">​</a></h3>
<p><code>matchType</code> field indicates a non-standard order fill.</p>
<p>When your order is filled by SOR, you will see <code>matchType: ONE_PARTY_TRADE_REPORT</code>, indicating that you did not trade directly on the exchange (<code>tradeId: -1</code>). Instead your order is filled by <em>allocations</em>.</p>
<p><code>allocId</code> field identifies the allocation so that you can query it later.</p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="what-are-allocations">What are allocations?<a class="hash-link" aria-label="Direct link to What are allocations?" title="Direct link to What are allocations?" href="/docs/binance-spot-api-docs/faqs/sor_faq#what-are-allocations">​</a></h3>
<p><strong>An allocation</strong> is a transfer of an asset from the exchange to your account. For example, when SOR takes liquidity from eligible order books, your order is filled by allocations. In this case you don&#x27;t trade directly, but rather receive allocations from SOR corresponding to the trades made by SOR on your behalf.</p>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="how-do-i-query-orders-that-used-sor">How do I query orders that used SOR?<a class="hash-link" aria-label="Direct link to How do I query orders that used SOR?" title="Direct link to How do I query orders that used SOR?" href="/docs/binance-spot-api-docs/faqs/sor_faq#how-do-i-query-orders-that-used-sor">​</a></h3>
<p>You can find them the same way you query any other order. The main difference is that in the response for an order that used SOR there are two extra fields: <code>usedSor</code> and <code>workingFloor</code>.</p>
<h3 class="anchor anchorWithStickyNavbar_fMI7" id="how-do-i-get-details-of-my-fills-for-orders-that-used-sor">How do I get details of my fills for orders that used SOR?<a class="hash-link" aria-label="Direct link to How do I get details of my fills for orders that used SOR?" title="Direct link to How do I get details of my fills for orders that used SOR?" href="/docs/binance-spot-api-docs/faqs/sor_faq#how-do-i-get-details-of-my-fills-for-orders-that-used-sor">​</a></h3>
<p>When SOR orders trade against order books other than the symbol submitted with the order, the order is filled with an <strong>allocation</strong> and not a trade. Orders placed with SOR can potentially have both allocations and trades.</p>
<p>In the API response, you can review the <code>fills</code> fields. Allocations have an <code>allocId</code> and <code>&quot;matchType&quot;: &quot;ONE_PARTY_TRADE_REPORT&quot;</code>, while trades will have a non-negative <code>tradeId</code>.</p>
<p>Allocations can be queried using <code>GET /api/v3/myAllocations</code> (Rest API) or <code>myAllocations</code> (WebSocket API).</p>
