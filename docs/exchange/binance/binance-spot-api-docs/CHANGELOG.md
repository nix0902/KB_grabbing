<p><strong>Clarification Regarding UTF-8 Encoding:</strong></p>
<ul>
<li>In <a href="/docs/binance-spot-api-docs/fix-api">FIX</a>, <a href="https://developers.binance.com/docs/binance-spot-api-docs/rest-api/general-api-information" target="_blank" rel="noopener noreferrer">REST</a>, and <a href="https://developers.binance.com/docs/binance-spot-api-docs/websocket-api/general-api-information" target="_blank" rel="noopener noreferrer">WebSocket APIs</a>, if your request contains a symbol name containing non-ASCII characters, then the response may contain non-ASCII characters encoded in UTF-8.</li>
<li>In REST and WebSocket APIs, some endpoints/methods may return asset and/or symbol names containing non-ASCII characters encoded in UTF-8 even if the request did not contain non-ASCII characters.</li>
<li>In <a href="/docs/binance-spot-api-docs/web-socket-streams">WebSocket Streams</a>, if your request contains a symbol name containing non-ASCII characters, then the stream events may contain non-ASCII characters encoded in UTF-8.</li>
<li>In WebSocket Streams, <a href="/docs/binance-spot-api-docs/web-socket-streams#all-market-mini-tickers-stream">All Market Mini Tickers Stream</a> and <a href="/docs/binance-spot-api-docs/web-socket-streams#all-market-rolling-window-statistics-streams">All Market Rolling Window Statistics Streams</a> events may contain non-ASCII characters encoded in UTF-8.</li>
<li>In <a href="/docs/binance-spot-api-docs/sbe-market-data-streams">SBE Market Data Streams</a>, if your request contains a symbol name containing non-ASCII characters, then the stream events may contain non-ASCII characters encoded in UTF-8.</li>
<li><a href="/docs/binance-spot-api-docs/user-data-stream">UserDataStream events</a> may contain non-ASCII characters encoded in UTF-8 if you own or trade any assets or symbols whose names contain non-ASCII characters.</li>
<li>For full compatibility with Binance APIs, please ensure your code is designed to handle UTF-8-encoded strings.</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2025-05-22">2025-05-22<a class="hash-link" aria-label="Direct link to 2025-05-22" title="Direct link to 2025-05-22" href="/docs/binance-spot-api-docs/CHANGELOG#2025-05-22">​</a></h3>
<p><strong>Notice: The following changes will happen at 2025-06-06 7:00 UTC.</strong></p>
<ul>
<li>The previous behavior of <code>recvWindow</code> on FIX, REST, and WebSocket APIs will be augmented by an additional check.
<ul>
<li>To review, the existing behavior is:
<ul>
<li>If <code>timestamp</code> is greater than <code>serverTime</code> + 1 second at receipt of the request, the request is rejected. Rejection by this check increments message limits (FIX API) and IP limits (REST and WebSocket APIs), but not Unfilled Order Count (order placement endpoints of all APIs).</li>
<li>If the difference between <code>timestamp</code> and <code>serverTime</code> at receipt of the request is greater than <code>recvWindow</code>, the request is rejected. Rejection by this check increments message limits (FIX API) and IP limits (REST and WebSocket APIs) but not Unfilled Order Count (order placement endpoints of all APIs).</li>
</ul>
</li>
<li>The additional check is:
<ul>
<li>Just before a request is forwarded to the Matching Engine, if the difference between <code>timestamp</code> and the current <code>serverTime</code> is greater than <code>recvWindow</code>, the request is rejected. Rejection by this check increments message limits (FIX API), IP limits (REST and WebSocket APIs), and Unfilled Order Count (order placement endpoints of all APIs).</li>
</ul>
</li>
<li>The documentation for Timing security has been updated to reflect the additional check.
<ul>
<li><a href="/docs/binance-spot-api-docs/rest-api/request-security#timing-security">REST API</a></li>
<li><a href="/docs/binance-spot-api-docs/websocket-api/request-security#timing-security">WebSocket API</a></li>
<li><a href="/docs/binance-spot-api-docs/fix-api#timing-security">FIX API</a></li>
</ul>
</li>

<li>Instead of expiring only the maker, only the taker, or unconditionally both orders, STP decrement decreases the available quantity of <strong>both</strong> orders and increases the <code>prevented quantity</code> of <strong>both</strong> orders by the amount of the prevented match.</li>
<li>This expires the order with less available quantity as (<code>filled quantity</code> + <code>prevented quantity</code>) equals <code>order quantity</code>. Both orders expire if their available quantities are equal. It is called a &quot;decrement&quot; because it reduces available quantity.</li>
</ul>
</li>
<li>Behavior when querying and/or canceling with <code>orderId</code> and <code>origClientOrderId/cancelOrigClientOrderId</code>:
<ul>
<li>The behavior when both parameters were provided was not consistent across all endpoints.</li>
<li>Moving forward, when both parameters are provided, the order is first searched for using its <code>orderId</code>, and if found, <code>origClientOrderId</code>/<code>cancelOrigClientOrderId</code> is checked against that order. If both conditions pass, the request succeeds. If both conditions are not met the request is rejected.</li>

<ul>
<li><code>GET /api/v3/order</code></li>

</li>
</ul>
</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2022-04-13">2022-04-13<a class="hash-link" aria-label="Direct link to 2022-04-13" title="Direct link to 2022-04-13" href="/docs/binance-spot-api-docs/CHANGELOG#2022-04-13">​</a></h3>
<p>REST API</p>
<ul>
<li>Trailing Stops have been enabled.
<ul>
<li>This is a type of algo order where the activation is based on a percentage of a price change in the market using the new parameter <code>trailingDelta</code>.</li>
<li>This can only used with any of the following order types: <code>STOP_LOSS</code>, <code>STOP_LOSS_LIMIT</code>, <code>TAKE_PROFIT</code>, <code>TAKE_PROFIT_LIMIT</code>.</li>
<li>The <code>trailingDelta</code> parameter will be done in Basis Points or BIPS.
<ul>
<li>For example: a STOP_LOSS SELL order with a <code>trailingDelta</code> of 100 will trigger after a price decrease of 1% from the highest price after the order is placed. (100 / 10,000 =&gt; 0.01 =&gt; 1%)</li>
</ul>
</li>
<li>When used in combination with OCO Orders, the <code>trailingDelta</code> will determine when the contingent leg of the OCO will trigger.</li>
<li>When <code>trailingDelta</code> is used in combination with <code>stopPrice</code>, once the <code>stopPrice</code> condition is met, the trailing stop starts tracking the price change from the <code>stopPrice</code> based on the <code>trailingDelta</code> provided.</li>
<li>When no <code>stopPrice</code> is sent, the trailing stop starts tracking the price changes from the last price based on the <code>trailingDelta</code> provided.</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_fMI7" id="2019-08-15">2019-08-15<a class="hash-link" aria-label="Direct link to 2019-08-15" title="Direct link to 2019-08-15" href="/docs/binance-spot-api-docs/CHANGELOG#2019-08-15">​</a></h3>
<p>REST API</p>
<ul>
<li>
<p>New order type: OCO (&quot;One Cancels the Other&quot;)</p>
<ul>
<li>
<p>An OCO has 2 orders: (also known as legs in financial terms)</p>
<ul>
<li><code>STOP_LOSS</code> or <code>STOP_LOSS_LIMIT</code> leg</li>
<li><code>LIMIT_MAKER</code> leg</li>
</ul>
</li>
<li>
<p>Price Restrictions:</p>
<ul>
<li><code>SELL Orders</code> : Limit Price &gt; Last Price &gt; Stop Price</li>
<li><code>BUY Orders</code> : Limit Price &lt; Last Price &lt; Stop Price</li>
<li>As stated, the prices must &quot;straddle&quot; the last traded price on the symbol. EX: If the last price is 10:
<ul>
<li>A SELL OCO must have the limit price greater than 10, and the stop price less than 10.</li>
<li>A BUY OCO must have a limit price less than 10, and the stop price greater than 10.</li>
</ul>
</li>
</ul>
</li>
<li>
<p>Quantity Restrictions:</p>
<ul>
<li>Both legs must have the <strong>same quantity</strong>.</li>
<li><code>ICEBERG</code> quantities however, do not have to be the same.</li>
</ul>
</li>
<li>
<p>Execution Order:</p>
<ul>
<li>If the <code>LIMIT_MAKER</code> is touched, the limit maker leg will be executed first BEFORE canceling the Stop Loss Leg.</li>
<li>if the Market Price moves such that the <code>STOP_LOSS</code> or <code>STOP_LOSS_LIMIT</code> will trigger, the Limit Maker leg will be canceled BEFORE executing the <code>STOP_LOSS</code> Leg.</li>
</ul>
</li>
<li>
<p>Canceling an OCO</p>
<ul>
<li>Canceling either order leg will cancel the entire OCO.</li>
<li>The entire OCO can be canceled via the <code>orderListId</code> or the <code>listClientOrderId</code>.</li>
</ul>
</li>
<li>
<p>New Enums for OCO:</p>
<ol>
<li><code>ListStatusType</code>
<ul>
<li><code>RESPONSE</code> - used when ListStatus is responding to a failed action. (either order list placement or cancellation)</li>
<li><code>EXEC_STARTED</code> - used when an order list has been placed or there is an update to a list&#x27;s status.</li>
<li><code>ALL_DONE</code> - used when an order list has finished executing and is no longer active.</li>
</ul>
</li>
<li><code>ListOrderStatus</code>
<ul>
<li><code>EXECUTING</code> - used when an order list has been placed or there is an update to a list&#x27;s status.</li>
<li><code>ALL_DONE</code> - used when an order list has finished executing and is no longer active.</li>
<li><code>REJECT</code> - used when ListStatus is responding to a failed action. (either order list placement or cancellation)</li>
</ul>
</li>
<li><code>ContingencyType</code>
<ul>
<li><code>OCO</code> - specifies the type of order list.</li>
</ul>
</li>
</ol>
</li>
<li>
<p>New Endpoints:</p>
<ul>
<li>POST api/v3/order/oco</li>

<li>GET api/v3/orderList</li>
</ul>
</li>
</ul>
</li>
<li>
<p><code>recvWindow</code> cannot exceed 60000.</p>
</li>
<li>
<p>New <code>intervalLetter</code> values for headers:</p>
<ul>

</li>
<li>
<p>New Header <code>X-MBX-ORDER-COUNT-(intervalNum)(intervalLetter)</code>that is updated on any valid order placement and tracks your current order count for the interval; rejected/unsuccessful orders are not guaranteed to have <code>X-MBX-ORDER-COUNT-**</code> headers in the response.</p>
<ul>

<li>New filters: <code>PERCENT_PRICE</code>, <code>MARKET_LOT_SIZE</code>, <code>MAX_NUM_ICEBERG_ORDERS</code>.</li>
<li>Added <code>RAW_REQUESTS</code> rate limit. Limits based on the number of requests over X minutes regardless of weight.</li>
<li>/api/v3/ticker/price increased to weight of 2 for a no symbol query.</li>
<li>/api/v3/ticker/bookTicker increased weight of 2 for a no symbol query.</li>
<li>DELETE /api/v3/order will now return an execution report of the final state of the order.</li>
<li><code>MIN_NOTIONAL</code> filter has two new parameters: <code>applyToMarket</code> (whether or not the filter is applied to MARKET orders) and <code>avgPriceMins</code> (the number of minutes over which the price averaged for the notional estimation).</li>
<li><code>intervalNum</code> added to /api/v1/exchangeInfo limits. <code>intervalNum</code> describes the amount of the interval. For example: <code>intervalNum</code> 5, with <code>interval</code> minute, means &quot;every 5 minutes&quot;.</li>
</ul>
<h4 class="anchor anchorWithStickyNavbar_fMI7" id="explanation-for-the-average-price-calculation">Explanation for the average price calculation:<a class="hash-link" aria-label="Direct link to Explanation for the average price calculation:" title="Direct link to Explanation for the average price calculation:" href="/docs/binance-spot-api-docs/CHANGELOG#explanation-for-the-average-price-calculation">​</a></h4>
<ol>
<li>
<p>(qty * price) of all trades / sum of qty of all trades over previous 5 minutes.</p>
</li>
<li>
<p>If there is no trade in the last 5 minutes, it takes the first trade that happened outside of the 5min window.
For example if the last trade was 20 minutes ago, that trade&#x27;s price is the 5 min average.</p>
</li>
<li>
<p>If there is no trade on the symbol, there is no average price and market orders cannot be placed.
On a new symbol with <code>applyToMarket</code> enabled on the <code>MIN_NOTIONAL</code> filter, market orders cannot be placed until there is at least 1 trade.</p>
</li>
<li>
<p>The current average price can be checked here: <code>https://api.binance.com/api/v3/avgPrice?symbol=&lt;symbol&gt;</code>
For example:
<a href="https://api.binance.com/api/v3/avgPrice?symbol=BNBUSDT" target="_blank" rel="noopener noreferrer">https://api.binance.com/api/v3/avgPrice?symbol=BNBUSDT</a></p>
</li>
</ol>
<p>USER DATA STREAM</p>
<ul>
<li><code>Last quote asset transacted quantity</code> (as variable <code>Y</code>) added to execution reports. Represents the <code>lastPrice</code> * <code>lastQty</code> (<code>L</code> * <code>l</code>).</li>
</ul>

<li>GET api/v3/openOrders with no symbol weight reduced to 40.</li>
<li>GET api/v3/ticker/24hr with no symbol weight reduced to 40.</li>
<li>Max amount of trades from GET /api/v1/trades increased to 1000.</li>
<li>Max amount of trades from GET /api/v1/historicalTrades increased to 1000.</li>

<li><code>REQUESTS</code> rate limit type changed to <code>REQUEST_WEIGHT</code>. This limit was always logically request weight and the previous name for it caused confusion.</li>
</ul>
<p>USER DATA STREAM</p>
<ul>
<li><code>cummulativeQuoteQty</code> field added to order responses and execution reports (as variable <code>Z</code>). Represents the cummulative amount of the <code>quote</code> that has been spent (with a <code>BUY</code> order) or received (with a <code>SELL</code> order). Historical orders will have a value &lt; 0 in this field indicating the data is not available at this time. <code>cummulativeQuoteQty</code> divided by <code>cummulativeQty</code> will give the average price for an order.</li>
<li><code>O</code> (order creation time) added to execution reports</li>
</ul>
